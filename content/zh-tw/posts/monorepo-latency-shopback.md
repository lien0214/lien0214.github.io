## 背景

ShopBack 使用 TypeScript monorepo。在我的實習期間，一個積極推進的遷移工作是把舊有的跨服務 API 呼叫——服務之間的 HTTP 呼叫——整合成 monorepo 內的直接函式呼叫。

這件事在紙面上聽起來是純基礎設施工作，沒有產品影響。在實際執行上，它改變了我對延遲的看法。

---

## 遷移前的狀態

相關服務透過 HTTP 互相呼叫，像這樣：

```typescript
// 遷移前：跨服務 HTTP 呼叫
const response = await fetch(`${INTERNAL_SERVICE_URL}/api/v1/user-watchlist/${userId}`);
const data = await response.json();
```

單獨看沒問題。但在一個串接 3–4 個這類呼叫的請求路徑中（在推薦和通知流程中很常見），你看到的是：

- 每次呼叫 ~20ms 的網路來回（即使在同一個資料中心內）
- 每次呼叫 ~15ms 的序列化/反序列化開銷
- 每次呼叫 ~10ms 的連線池管理和 TLS 開銷
- 每個跳躍點的錯誤處理表面積
- 分散式追蹤噪音

3 個串接呼叫 × 每次 ~50ms = 150ms 的附帶延遲，加在實際業務邏輯之上。

---

## 遷移後的狀態

monorepo 遷移後，函式可以直接 import：

```typescript
// 遷移後：monorepo 內的直接函式呼叫
import { getUserWatchlist } from '@shopback/user-service';

const watchlist = await getUserWatchlist(userId);
```

每次呼叫現在在 10ms 以內。業務邏輯完全相同。網路消失了。

---

## 為什麼這比看起來更重要

工程面的論點很明顯：更少的基礎設施開銷、更小的失敗表面積、更容易除錯。讓我驚訝的是產品面的論點。

**用戶感受得到 100ms。** 這在 HCI 研究中有實證記錄，從 Miller（1968）一路到現代 UX 研究都在確認。300ms 的回應感覺「即時」。600ms 感覺「快」。1000ms 感覺「慢」。分界線大約在互動式 UI 的 400–600ms 附近。

現在，一個有 4 個串接服務呼叫（每個 ~50ms）的功能，在任何業務邏輯跑之前就有 200ms 的基礎開銷。業務邏輯再加 100–200ms。在用戶瀏覽器的網路之前，你已經在 300–400ms 了。

遷移後，開銷是 ~40ms（直接呼叫）。業務邏輯相同。你在 140–240ms。你從「快速連線上明顯感覺慢」移動到了「一般連線上感覺快」。

我在這個遷移發生的功能上看到的 +30% 用戶互動行為提升，如果你相信延遲與互動的文獻，是合理的。用戶不會有意識地察覺速度差異。他們就是互動得更多。

---

## 實際要做的事

這個遷移不是簡單的搜尋取代。複雜之處在於：

**依賴方向很重要。** 如果服務 A 透過 HTTP 呼叫服務 B，A 在編譯時不依賴 B。遷移後它會依賴。這可能在 monorepo 的依賴圖中造成建置系統不允許的循環依賴。你要在開始之前先理解依賴結構。

**錯誤處理改變語意。** HTTP 呼叫可能因為網路錯誤（連線被拒、逾時）失敗，這和函式呼叫拋出異常是截然不同的類別。遷移後，如果不小心處理，你會失去「網路錯誤 vs. 邏輯錯誤」的區別。我們明確定義了錯誤變體的型別，並相應更新了錯誤處理邏輯。

**可觀測性改變。** HTTP 呼叫有自動的分散式追蹤 Span。函式呼叫沒有。我們在熱路徑上加入了明確的遙測插樁以維持可觀測性。

**測試需要更新。** 任何在網路層 Mock HTTP 呼叫的測試，都需要改成在函式介面層 Mock。這在大多數情況下是更好的測試（你現在測試的是你實際使用的介面），但它是工作量。

---

## 更廣泛的教訓

留在我腦海中的事：**效能提升的原因是結構性的，不是演算法性的。** 我們沒有優化任何業務邏輯。沒有調整任何資料庫查詢。沒有加快取。我們只是移除了一個因歷史原因而存在、而非技術原因的架構接縫。

在規模下，許多「低垂的果實」效能工作看起來都像這樣。程式碼沒問題。資料庫查詢沒問題。瓶頸在於系統的*形狀*——服務之間劃定的邊界、它們之間的合約、那些合約的開銷。

理解這個區別——演算法優化 vs. 架構優化——是我從這份實習得到的最有用的東西之一。
